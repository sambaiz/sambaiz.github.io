DuckDB の Hive パーティションプルーニングの仕組みをソースコードから理解する

2026-02-15 database

DuckDB で Hive パーティションを読み込む際、WHERE 条件に基づいて不要なファイルをスキップするプルーニングが行われる。この仕組みをソースコードから理解する。

$ duckdb --version

プルーニングの例

Hive パーティション形式のデータを用意する。

$ duckdb
D COPY (SELECT i, i % 12 + 1 AS month FROM range(1000000) t(i))
  TO 'data' (FORMAT PARQUET, PARTITION_BY (month));

hive_partitioning=true を指定して読み込む。

D EXPLAIN ANALYZE
  SELECT * FROM read_parquet('data/**/*.parquet', hive_partitioning=true)
  WHERE month = 1;

12 ファイル中 1 ファイルのみスキャンしている。

DuckDB のクエリ実行は Parser → Binder → Optimizer → Executor の順に進む。プルーニングは Optimizer と Executor の両方で行われる。

Optimizer でのプルーニング

Binder は Parser が生成した AST を受け取り、テーブル関数の引数を解析する。この際、ファイルパスから Hive パーティション情報を抽出する。HivePartitioning::Parse() がパス文字列から key=value のペアを解析し、MultiFileReader::BindOptions() が bind_data.hive_partitioning_indexes に登録する。

read_parquet() のようなテーブル関数はフィルタ処理を行う pushdown_complex_filter というコールバックを実装していて、FilterPushdown::PushdownGet() がこれにフィルタ式を渡す。

HivePartitioning::ApplyFiltersToFileList() はファイルごとに次の処理を行って読むファイルを絞り込む。

  1. ファイルパスからパーティション値を取得: data/month=2/... → month=2
  2. ConvertKnownColRefToConstants() でフィルタ式内のカラム参照を定数に置換: month → 2
  3. 式を評価: WHERE month = 1 → 2 = 1 → false
  4. false ならファイルを除外

JOIN を含むクエリでは、JOIN 条件にパーティションカラムが含まれていれば片方のテーブルのカラムのみでフィルタした場合でも両テーブルにプルーニングが効く。 これは FilterCombiner が値の等しいカラムをグルーピングして同様に扱うため。ON t1.month = t2.month と WHERE t1.month = 1 から、両テーブルに month = 1 が適用される。

D SELECT *
  FROM read_parquet('data1/**/*.parquet', hive_partitioning=true) t1
  JOIN read_parquet('data2/**/*.parquet', hive_partitioning=true) t2 ON t1.month = t2.month
  WHERE t1.month = 1;

Executor でのプルーニング

サブクエリは HASH_JOIN に変換され、その結果の min/max 外のファイルをスキップする Join Filter Pushdown が適用される。なお BigQuery ではサブクエリのような動的な式では全パーティションがスキャンされる。

D SELECT *
  FROM read_parquet('data/**/*.parquet', hive_partitioning=true) t1
  WHERE t1.month = (SELECT month FROM other_table WHERE key = 1);
