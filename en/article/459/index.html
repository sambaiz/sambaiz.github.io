Setting security groups for EKS cluster's nodes created with CDK

2023-12-05 awskubernetes

aws_eks.Cluster in CDK has a securityGroup field, but this is an SG associated with the ENI of the control plane, not the node. If it is not passed, it will be automatically created.

Then, where is the SG settings for EKS cluster’s nodes? It is in the node group’s auto scaling group.

When you create an EKS cluster with CDK and DefaultCapacityType is NODEGROUP, which is the default, manged node group is created, which consists of defaultCapacity nodes and EKS performs drain etc.

However, even if you access defaultCapacity field of the cluster to refer to the managed node group, nothing will happen because this field only has a value when DefaultCapacityType is EC2. In the case of NODEGROUP, you should use defaultNodegroup instead, but unlike the former, which returns AutoScalingGroup, it returns Nodegroup, and it doesn’t seem to have an interface like addSecurityGroup().

So, I set defaultCapacity to 0 to prevent the node group from being created by default, and added SG with addAutoScalingGroupCapacity() that returns AutoScalingGroup.

const cluster = new eks.Cluster(this, 'Cluster', {
  ...
  defaultCapacity: 0,
})

const securityGroup = new ec2.SecurityGroup(this, 'EKSClusterDefaultCapacitySecurityGroup', {
  vpc,
  allowAllOutbound: true,
})

const defaultCapacityAsg = cluster.addAutoScalingGroupCapacity('EKSClusterDefaultCapacity', {
  instanceType: new ec2.InstanceType('t3.small'),
  minCapacity:  2,
})

defaultCapacityAsg.connections.addSecurityGroup(securityGroup)

Be careful if you add an existing SG. If the SG has duplicate settings with the automatically added ones, they will be deleted together when the resource is deleted. Therefore, I think it is better to create a new one instead of reusing it.
